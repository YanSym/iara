1. Sumário executivo
Este documento consolida o handoff técnico para a empresa terceirizada responsável por implementar o novo agente IAra multi-tenant.

A entrega esperada é um runtime Python modular, multi-tenant, orquestrado por LangGraph, com entrada por webhooks, validação Pydantic, controle operacional por RabbitMQ + Postgres, integração com Chatwoot via MCP como adapter operacional de provider/plataforma, Agent Tools MCP governadas para ferramentas de negócio, catálogo de MCPs por tenant, configuração publicada em Postgres, memória governada, segurança fail-closed, auditoria sanitizada e critérios claros de aceite.

Esta contratação não é uma implementação de fluxo único N8n, não é um protótipo sem governança e não é um produto SaaS completo. O foco é o agente/runtime e suas integrações seguras.

Resultado esperado ao final:

Agente conversacional multi-tenant operando por grafo LangGraph.
Contratos Pydantic para entrada, estado, tools, comandos, respostas, auditoria e evidências.
ProviderAdapter para isolar Chatwoot e futuros providers.
ChatwootMcpAdapter/ChatwootMcpRegistry para chamadas MCP operacionais de Chatwoot por account/tenant.
AgentToolMcpGateway/AgentToolRegistry para tools lógicas de negócio, com schema, policy, skill, status e testes.
Catálogo tenant-scoped de servidores MCP disponíveis, solicitados, em sandbox, ativos ou suspensos.
Persistência operacional em Postgres e filas RabbitMQ para backpressure/redelivery.
Outbox/readback/idempotência para todo side effect externo.
Dashboard/config organizer como superfície administrativa de draft e publicação segura, se incluído na etapa contratada.
Evidências sanitizadas e documentação suficiente para operação, manutenção, rollback e auditoria.



2. Princípios obrigatórios
2.1. LangGraph é o baseline de orquestração
A implementação deve modelar o fluxo como grafo LangGraph, com:

State explícito.
Nodes nomeados e testáveis.
Subgraphs para mídia, tools, kanban, CommandAssistant e rotinas especializadas.
Edges condicionais documentadas.
Checkpoints para retomada de execução.
Retries por node, com idempotência externa.
Interrupções/HITL para decisões humanas quando necessário.
Versionamento de flow/config/prompt/tool policy por execução.

Pipeline simples só pode existir como harness local, protótipo sem side effects ou lógica interna de um node. Ele não substitui o contrato arquitetural LangGraph.
2.2. Python modular e testável
O grafo não deve esconder regra de negócio. Cada node deve chamar módulos Python isolados, com testes unitários e de integração fake/stub.

Stack técnica recomendada:

FastAPI para entrypoints HTTP/webhooks.
Pydantic v2 para schemas, validação, aliases, serialização e redaction.
LangGraph para orquestração stateful.
RabbitMQ para transporte, backpressure e redelivery.
Postgres para idempotência, leases, debounce, outbox, auditoria, configuração, memória semântica governada e checkpoints.
2.3. Chatwoot é provider/plataforma, não canal
Nomenclatura obrigatória:

provider/platform: sistema externo integrado pelo runtime, por exemplo Chatwoot.
source_channel ou inbox_channel_type: canal real dentro do Chatwoot, por exemplo WhatsApp, Instagram, Facebook/Messenger, Email, Website, API etc.
inbox_id: instância concreta do canal dentro da account Chatwoot.
ProviderAdapter: interface que adapta um provider/plataforma ao runtime.
ChatwootMcpAdapter: implementação do ProviderAdapter para operar Chatwoot via MCP.
ChatwootMcpRegistry: registry explícita de capabilities reais do Chatwoot MCP por tenant/account.

O LLM nunca deve receber catálogo bruto de MCP Chatwoot nem decidir nome real de tool MCP. O runtime deve resolver capabilities por registry explícita e policy.
2.4. Duas camadas MCP distintas
Camada 1 — MCP operacional de provider/plataforma:
Exemplo principal: Chatwoot MCP.
Uso interno pelo ProviderAdapter.
Não exposto livremente ao agente.
Protegido por registry, allowlist, denylist, risk class, account binding, idempotência e readback.
Camada 2 — Agent Tools MCP:
Ferramentas de negócio do agente, expostas por nomes lógicos.
Exemplos: disponibilidade, agendar, qualificado, desqualificado, kanban_analisar_conversa, campanha_criar_rascunho, campanha_validar_publico, campanha_disparar_lote, followup_reengajar_conversa, kb_sugerir_atualizacao, historico_analisar_conversas, voice_responder_audio.
Cada tool exige schema, descrição, skill de uso, policy, status, binding de backend e testes.
2.5. Fail-closed e anti-cross-tenant
Qualquer dúvida de tenant/account/inbox/source_channel/capability deve bloquear a operação. Não deve haver fallback permissivo, inferência por similaridade nem execução em account errada.


3. Escopo contratado
Entram no escopo de contratação do agente/runtime:
3.1. Atendimento conversacional
Receber eventos de conversa.
Filtrar eventos não elegíveis.
Construir contexto seguro.
Responder ao lead conforme prompt/config publicada.
Acionar handoff humano quando necessário.
Aplicar anti-loop, baixa confiança e guardrails.
3.2. Entrada, normalização e elegibilidade
Webhook HTTP/FastAPI.
Tenant resolver.
Validação de headers/body.
NormalizedChatwootEvent.
EligibilityDecision.
Cálculo de correlation_id, idempotency_key e raw_hash.
Rejeição/ignorar eventos outgoing, bot/system, private ou fora de escopo.
Enriquecimento read-only controlado quando faltar campo essencial.
3.3. Controle operacional
RabbitMQ para jobs, wakeups, backpressure e redelivery.
Postgres como fonte de verdade para event_receipts, conversation_debounce, conversation_run_leases, provider_command_outbox, agent_runs, runtime_run_steps, runtime_errors, auditoria e checkpoints.
Debounce por conversa.
Lease/fencing token para evitar concorrência.
Idempotência por evento, comando e destinatário.
DLX/retry/backoff.
Readback antes de marcar side effect como concluído.
3.4. Mídia e arquivos
Subgraph MediaUnderstanding antes do agente principal.
Áudio: transcrição detalhada.
Imagens/documentos escaneados: OCR/descrição visual quando configurado.
PDFs/DOCX/PPTX/XLSX/texto: extração textual quando suportado.
Fallback explícito: partial, unsupported ou failed.
Arquivo bruto, base64, URL temporária e headers não entram no State, prompt, log ou evidência.
3.5. Contexto, memória, prompt e KB
ConversationContext limpo e redigido.
Histórico recente permitido apenas como contexto operacional, não memória semântica automática.
Prompt/config somente de versões publicadas.
KB consultiva segmentada e segura.
Memória semântica governada por namespace, TTL, consentimento, retenção, purge/anonymize e redaction.
Checkpoint LangGraph separado de memória semântica.
3.6. Agent Tools MCP
AgentToolRegistry.
AgentToolMcpGateway.
ToolPolicyGuard.
ToolSkillResolver.
ToolExecutor.
ToolInvocationRequest/ToolInvocationResult.

3.7. Agenda e sistemas de domínio
Google Agenda/Calendar como managed/default possível.
Clinicorp como managed/default possível.
Adapter próprio/custom por tenant quando solicitado e aprovado.
Consultar disponibilidade antes de confirmar agendamento.
Escritas com idempotência, outbox, readback e fallback/handoff em ambiguidade.
3.8. Qualificação, handoff e notificações
Qualificação como operação composta: critério, label permitida, nota privada segura, notificação interna configurada e readback.
Handoff humano com motivo, contexto seguro e sem vazar prompt/memória interna.
Falha parcial deve ser registrada de forma explícita e segura.
3.9. Kanban operacional
ConversationKanbanSubagent como subgraph/tool governada.
Primeira entrega recomendada: suggest_only.
Escritas em Chatwoot só com kanban_policy publicada, outbox, idempotência e readback.
Status nativo do Chatwoot não deve ser confundido automaticamente com etapa de negócio.
Comentários de kanban devem ser notas privadas seguras, nunca mensagens públicas.
3.10. CommandAssistant e campanhas/outbound
CommandAssistantSubgraph separado do atendimento ao lead.
Comandos administrativos apenas para autores/números autorizados por CommandRequesterBinding.
Telefone bruto não é autorização suficiente.
Campanhas são high-risk.
Default seguro: draft_only/dry_run.
Envio real apenas em approved_send, com policy, opt-out/consentimento, template/canal compatível, rate limit, HITL quando exigido, outbox por destinatário, idempotência e readback.
Evidência nunca pode conter lista integral de contatos.
3.11. Follow-up individual
Follow-up em conversa existente, separado de campanha em massa.
Respeitar opt-out, quiet hours, tentativas máximas, contexto suficiente e policy por tenant.
Side effect por outbox/readback.
3.12. Análise histórica e atualização de KB
Análise histórica ampla somente quando autorizada.
Modo read-only/sensitive, com redaction.
Não persistir conversa bruta por padrão.
Saídas devem virar drafts revisáveis: KnowledgeBootstrapDraft, PromptConfigDraft, KbUpdateDraft.
KB via WhatsApp ou comando operacional nunca publica direto; gera draft para revisão/HITL.
3.13. Voz/áudio outbound
voice_responder_audio governado por voice_output_policy.
Provider/modelo/voz/credential_ref no runtime, nunca no prompt.
Guardrails antes de TTS.
Fallback obrigatório para texto.
Sem persistir áudio bruto em evidência.
3.14. Catálogo de MCPs por tenant
tenant_mcp_servers como catálogo tenant-scoped.
MCPs managed/default possíveis: Chatwoot, Google Agenda, Clinicorp.
MCP customizado do cliente entra por request, sandbox/read-only, discovery, risk classification, mapping, policy/skill, QA/HITL e publicação transacional.
Prompt final recebe apenas tools lógicas ativas; nunca catálogo bruto de servidores/tools MCP.
3.15. Dashboard e Agent Config Organizer
Dashboard App externo no Chatwoot pode ser superfície administrativa da primeira entrega.
Chatwoot iframe não deve receber tokens nem publicar produção diretamente.
Backend IAra valida origem, account, tenant, usuário e permissão.
Agent Config Organizer é grafo administrativo separado: cria drafts, simula, valida e prepara publicação.
PublishService deve ser determinístico e transacional.

5. Arquitetura-alvo em alto nível
Fluxo conceitual:

Webhook HTTP recebe evento.
Auth/tenant/account/inbox/source_channel são validados.
Evento bruto é referenciado por hash/ref seguro; payload bruto não é persistido por padrão.
Pydantic normaliza para NormalizedChatwootEvent.
EligibilityDecision aceita, ignora, rejeita ou pede enriquecimento read-only.
Idempotência e debounce são registrados em Postgres.
Wakeup/job é enviado ao RabbitMQ.
Worker consome job, faz claim de lease por conversa e inicia/retoma LangGraph.
MediaUnderstanding processa áudio/imagens/documentos antes do agente.
ConversationContext é montado com histórico permitido, memória governada, KB, config publicada e tools ativas.
Agente principal decide resposta e possíveis tool calls lógicas.
ToolPolicyGuard valida cada tool call.
Tools com side effect viram comandos/outbox, não chamadas diretas reexecutáveis.
Guardrails normalizam resposta e impedem vazamentos.
ProviderCommand é executado pelo ProviderAdapter/ChatwootMcpAdapter.
Readback confirma side effect.
Auditoria sanitizada registra status, hashes, refs e versões usadas.

8. Segurança, LGPD e redaction
8.1. Proibições absolutas
É proibido inserir em repositório, documento, log ou evidência:

tokens;
headers sensíveis;
cookies;
credenciais;
account IDs reais;
telefones brutos;
payload bruto de webhook;
pinData;
URLs temporárias;
anexos brutos;
áudio/imagem/base64;
dumps MCP completos;
lista integral de leads;
conteúdo integral de conversas;
prompt interno sensível;
chain-of-thought.
8.2. Regras obrigatórias
Credenciais somente por secret_ref/credential_ref.
Mensagens privadas não entram em prompt, memória, resposta ou evidência detalhada.
Dados pessoais/sensíveis exigem minimização, finalidade, TTL, redaction, opt-out e purge/anonymize.
Dados de saúde devem ser tratados como sensíveis.
Evidência deve usar hashes, contagens, refs seguras, status e erros sanitizados.
Cross-tenant deve falhar fail-closed antes de qualquer chamada externa.
Writes precisam de idempotência e readback.
Writes high-risk exigem policy e, quando aplicável, HITL.
Campanhas exigem consentimento/opt-out, janela/canal/template compatível, rate limit, aprovação e trilha de auditoria.
Produção não pode ser acessada sem autorização explícita da Digi2B.

G4 — Controle operacional
Aceite:

RabbitMQ + Postgres operando conforme baseline.
Testes de webhook duplicado, rajada, dois workers, redelivery, lease expirado e retomada LangGraph.
Side effects não duplicam após retry/checkpoint.
Outbox/readback implementados.
G5 — Agent Tools MCP e side effects sandbox
Aceite:

Apenas tools active aparecem ao agente.
Agenda consulta disponibilidade antes de agendar.
Qualificação e notificações respeitam policy.
Kanban inicia suggest_only ou modo aprovado.

KB update gera draft, não publicação direta.

G6 — Piloto controlado
Aceite:

Incoming válido gera uma resposta uma vez.
Outgoing/bot/private não aciona resposta.
Debounce, redelivery e concorrência não duplicam mensagens.
Mídia passa por MediaUnderstanding antes do agente.
Guardrails rodam antes da resposta.
Logs e evidências sem segredos/payload bruto.

11. Riscos técnicos principais
Cross-tenant/account mismatch: mitigar com binding explícito, registry e fail-closed.
Inferência de nomes MCP: proibida; usar ChatwootMcpRegistry.
Duplicidade de mensagens/side effects: mitigar com Postgres idempotency ledger, outbox, readback e fencing token.
Retomada LangGraph reexecutando ação externa: side effects nunca diretos em node reexecutável.
Debounce/concorrência por conversa: usar conversation_debounce e conversation_run_leases.
Mensagens privadas vazando: excluir do prompt/memória/evidência por padrão.
Payload bruto/URLs temporárias/anexos em logs: proibir e testar redaction.
Campanhas sem governança: exigir CommandRequesterBinding, policy, HITL, opt-out, rate limit, outbox e readback.
MCP customizado inseguro: sandbox, egress/roots restritos, discovery sanitizado e risk classification.
Memória longa sem política: começar mínima/desabilitada se política não estiver fechada.
Confundir status Chatwoot com kanban de negócio: usar kanban_policy por tenant.
Mídia não processada gerando resposta inventada: MediaUnderstanding obrigatório.
Dashboard virar fonte de verdade: Postgres publicado é a fonte canônica.

14. Nota de segurança para compartilhamento
Este documento foi preparado para compartilhamento externo com uma empresa terceirizada. Ele descreve arquitetura, contratos, responsabilidades e critérios de aceite sem incluir credenciais, tokens, dados reais de tenants, account IDs reais, telefones reais, payloads brutos, pinData, URLs temporárias ou dumps live.

Qualquer material adicional solicitado pela terceirizada deve seguir a mesma regra: fornecer somente amostras sintéticas ou evidências sanitizadas, salvo autorização explícita da Digi2B para um ambiente controlado.
