IAra é um runtime para agentes conversacionais LLM multi-tenant em SaaS. O fluxo é:

1. Webhook do Chatwoot → resolve tenant → normaliza evento → checa elegibilidade
2. RabbitMQ → JobConsumerWorker adquire lease → executa grafo LangGraph
3. LangGraph → 7 nós: eligibility → media → context → agent (LLM) → tools → guardrails → dispatch
4. Outbox Pattern → comands pendentes no Postgres → OutboxDrainerWorker executa via adapter → readback
5. Resposta → volta para o contato no Chatwoot

Infraestrutura: FastAPI + LangGraph + Anthropic/OpenAI + RabbitMQ + PostgreSQL + SQLAlchemy async.

Bugs e gargalos encontrados

Bug 1 — CRÍTICO: Comandos falhos nunca são reprocessados (outbox_drainer.py)

mark_failed() seta status="failed", mas fetch_pending_all() filtra apenas status="pending". Resultado: um comando que falha é abandonado para sempre. O código de max_retries/dead-letter nunca é executado — é código morto.

Bug 2 — CRÍTICO: build_production_graph não injeta gateway nem outbox (graph/builder.py)

build_production_graph só injeta o LLM. gateway=None e outbox_service=None fazem os nós tool_executor e command_dispatch usarem o modo stub de teste, retornando resultados falsos e nunca despachando mensagens ao Chatwoot.

Bug 3 — MODERADO: Eligibility aceita eventos sem message_type (eligibility/decision.py)

Eventos como conversation_created têm message_type=None. As regras 2 e 3 do checker só rejeitam OUTGOING/ACTIVITY/TEMPLATE — None passa por tudo e o agente é acionado sem mensagem real.

Bug 4 — MENOR: fetch_pending_all ignora scheduled_at para backoff

A query ordena por scheduled_at mas não filtra por ele. Quando o retry com backoff for implementado, o filtro precisa existir.

Bug 5 — MENOR: hash_for_ref em redaction.py é código morto

Definida mas nunca chamada no codebase inteiro.
